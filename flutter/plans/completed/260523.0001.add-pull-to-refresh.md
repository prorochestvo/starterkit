# Task Breakdown

## Overview

Add pull-to-refresh to the feed screen so users can manually trigger a reload without restarting the app. The feed currently only refreshes on first mount and on a hard navigation back to it, which is confusing when network had been down at mount time.

## Assumptions

- Feed state is managed by `FeedCubit` (BLoC pattern) — adding a `refresh()` method that re-emits `FeedLoading` and re-calls the repository.
- `RefreshIndicator` from Material is sufficient (no Cupertino variant needed yet — iOS uses the same widget set in this app).
- Network failure during refresh keeps existing list visible and surfaces an error snackbar; we do not blank the screen.

## Tasks

### Task 1: Add `FeedCubit.refresh()`
- Description: New public method that emits `FeedLoading` then calls `feedRepository.fetchLatest()`. Reuses the same success/error branches as the initial load.
- Acceptance Criteria:
  - `blocTest` covering: `refresh` emits `[FeedLoading, FeedLoaded]` on success.
  - `blocTest` covering: `refresh` emits `[FeedLoading, FeedError]` on repository failure.
- Pitfalls & edge cases: Don't drop in-flight requests — if `refresh()` is called twice quickly, the second call should win and the first should be ignored (use a request-id token).
- Complexity: Easy

### Task 2: Wrap `FeedListView` in `RefreshIndicator`
- Description: Existing `ListView.builder` wrapped in `RefreshIndicator` whose `onRefresh` returns the future from `FeedCubit.refresh()`.
- Acceptance Criteria:
  - Widget test: dragging down dispatches `refresh()` exactly once.
  - Widget test: returned future completes only after `FeedCubit` settles (not on `FeedLoading` emit).
- Pitfalls & edge cases: `RefreshIndicator` requires a scrollable child — empty-feed state currently uses `Center`, must keep the scrollable above the empty placeholder using `AlwaysScrollableScrollPhysics`.
- Complexity: Medium

### Task 3: Error snackbar on refresh failure
- Description: Listen for `FeedError` in the screen's `BlocListener`; show `SnackBar` with the rendered `Failure` message and a retry action that re-invokes `refresh()`.
- Acceptance Criteria:
  - Widget test: refresh failure causes a snackbar with the expected text from the `Failure` mapper.
  - Snackbar's retry button triggers `refresh()` again.
- Pitfalls & edge cases: Snackbars stack badly if the user pulls multiple times; clear the queue before showing the new one.
- Complexity: Easy

## Execution Order

1. Task 1 (cubit + tests)
2. Task 2 (widget integration + tests)
3. Task 3 (snackbar + tests)

## Risks

- `RefreshIndicator` interferes with the existing horizontal carousel inside each feed card. Mitigation: explicit `ScrollPhysics` on the carousel + manual gesture arena check; reviewer to look at this specifically.
- Re-fetching every pull may strain the API. Mitigation: rate-limit refresh on the cubit side (ignore calls within 2s window).

## Trade-offs

- Using Material `RefreshIndicator` instead of platform-adaptive — saves one dependency and a custom widget, costs the iOS-native bounce feel. Acceptable for now; revisit if iOS users complain.
