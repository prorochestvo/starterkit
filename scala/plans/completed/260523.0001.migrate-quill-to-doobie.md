# Task Breakdown

## Overview

Migrate the `UserRepository` and `OrderRepository` from Quill to Doobie. Quill's macro-driven query generation has been the source of three production bugs in the last quarter (mis-projected joins silently dropping columns); the team has decided to standardize on Doobie's explicit `Fragment`-based queries for the database access layer.

## Assumptions

- ZIO 2 is the effect runtime (already in place). Doobie integration via `zio-interop-cats` is already a transitive dependency.
- Test database is Postgres in docker-compose; integration tests live under `it/` and run with `sbt IntegrationTest/test`.
- This migration touches only the two repositories listed. The remaining `ProductRepository` stays on Quill until a follow-up (kept separate so the diff is reviewable).

## Tasks

### Task 1: Add Doobie + `HikariTransactor` wiring
- Description: New `Database` module exposing a `Transactor[Task]` via ZIO `ZLayer`. Uses HikariCP with config from the existing `database.conf`. Replaces the current Quill `PostgresZioJdbcContext`.
- Acceptance Criteria:
  - Unit test (`suite("Database")`) verifies the layer constructs without error against an embedded Postgres.
  - Existing app startup time does not regress (sanity bench, ±10%).
- Pitfalls & edge cases: Two transactors at once (Quill + Doobie) during migration will double-connection-pool. Cap each pool at half its previous size for the duration of the migration.
- Complexity: Medium

### Task 2: Port `UserRepository` to Doobie
- Description: Rewrite all queries (`findById`, `findByEmail`, `insert`, `update`, `softDelete`) as `Fragment`s with explicit column projections. Convert results via `Read[User]` typeclass derived through `Meta` instances.
- Acceptance Criteria:
  - Existing `UserRepositorySpec` passes unchanged (test surface is the public interface).
  - Integration spec (`UserRepositoryIT`) added covering the soft-delete path against real Postgres.
  - Typed error channel: SQL errors mapped to `DomainError.DatabaseUnavailable` or `DomainError.NotFound`; nothing leaks raw `SQLException`.
- Pitfalls & edge cases: Quill auto-generated the `created_at` / `updated_at` columns via macro; Doobie won't — explicitly set them in the `INSERT` fragment. The `users.deleted_at IS NULL` filter must be on every `find*` query (Quill applied it via implicit; Doobie won't).
- Complexity: Hard

### Task 3: Port `OrderRepository` to Doobie
- Description: Same shape as Task 2 for the order aggregate. Includes the `Order` ↔ `OrderItem` one-to-many join via `Fragment.const` + `Read[(Order, OrderItem)]` + `groupBy` in Scala.
- Acceptance Criteria:
  - Existing `OrderRepositorySpec` passes unchanged.
  - Integration spec verifies the join recovers the same data Quill produced (parity test against a checked-in fixture).
  - Pagination tests cover offset > 0 (Quill silently mis-paginated in one of the bugs that motivated this migration).
- Pitfalls & edge cases: The previous Quill query joined `orders` to `order_items` and projected the user_id from `orders` — but the macro reorders columns alphabetically in some cases, which is what caused the silent data swap last quarter. Be explicit about column order in `Read` derivation.
- Complexity: Hard

### Task 4: Remove Quill dependency from `build.sbt` for these two repos
- Description: Confirm nothing else imports `io.getquill.*` in `users` or `orders` modules; remove `quill-jdbc-zio` from their `libraryDependencies`. `ProductRepository` retains it.
- Acceptance Criteria:
  - `sbt compile` clean.
  - `sbt scalafmtCheckAll` clean.
- Complexity: Easy

## Execution Order

1. Task 1 (wiring) — unlocks Tasks 2 and 3.
2. Task 2 (Users) — simpler entity, validates the migration pattern.
3. Task 3 (Orders) — applies the pattern to the join-heavy aggregate.
4. Task 4 (cleanup) — only after 2 and 3 are merged and green.

## Risks

- Performance regression: Doobie's explicit Fragments may be slower than Quill's compile-time-optimized SQL on the hot path. Mitigation: benchmark `findById` and `listByCustomer` queries before/after; rollback if > 15% regression.
- Behavior parity: the parity test against a fixture is the strongest signal we have. If the fixture itself is wrong, parity is meaningless. Reviewer to sanity-check the fixture rows.

## Trade-offs

- Verbosity: Doobie queries are 3–5× longer than the equivalent Quill macro. Accepted in exchange for explicitness and the ability to grep for column names in queries (which Quill makes impossible).
- Migration window: keeping Quill and Doobie side-by-side for two PRs doubles the pool size and the build graph briefly. Accepted as the price of a reviewable diff.
